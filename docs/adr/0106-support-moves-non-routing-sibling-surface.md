# Support moves are a non-routing sibling surface fanned from the capability oracle

Support moves let a co-located participant declare a flavored action fanned
from *their own* gifts (capabilities via the ownership oracle, plus
predicate-tree legs for distinction/trait combos). A declaration takes the
place of the helper's pick/vote in the group flow. The helper rolls their
own check; success banks easing that enters the resolving check via
`perform_check(extra_modifiers=...)`; failure can fire a complication on
the helper via the existing consequence pipeline.

Density comes from a `MissionAssistPattern` catalog that auto-offers moves
wherever context matches; authored gems on specific nodes add to or
suppress patterns. Discovery is part of the fun: qualifying moves light up;
`rumored` moves show a veiled tease to the whole party; the journal shows
an unseen-approaches count after resolution.

> Status: accepted · Source: #2046

**Rejected alternatives:**
- *Silent bonus-lending* — helpers don't roll, just donate a stat. Rejected:
  helping should be drama, not a stat donation.
- *JOINT-only teamwork* — support only inside JOINT nodes. Rejected: the
  group flow has no "single-actor node with bystanders" state; any 2+ PC
  run resolves every node through the group machinery.
- *Bare-predicate qualifiers duplicating the capability layer* — using only
  predicate trees for qualification. Rejected: capabilities are the
  primary ownership oracle; predicate legs cover combos that aren't
  capabilities.
